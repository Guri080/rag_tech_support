# I Built a Fraud Pipeline with Tessera in an Afternoon. Here's What I Learned.

*By Marcus Okonkwo, Senior ML Engineer at a fintech I'm not going to name. Cross-posted to my personal blog.*

I have been on a personal mission to evaluate every feature store I can get my hands on, because the one we use at work is held together with duct tape and I am tired. Last Saturday I sat down with Tessera 2.0, a free Kafka cluster on Confluent Cloud, and a synthetic transactions generator I had written for a previous evaluation. By dinner I had a working real-time fraud feature pipeline. This post is the rough walkthrough.

I am not affiliated with Tessera. I paid for the trial license out of pocket. I am also going to point out the things that annoyed me, because that is more useful than a brochure.

## The setup

I had three sources of transaction events flowing into Kafka: a `transactions` topic with cardholder events, a `chargebacks` topic with confirmed fraud labels, and a `device_signals` topic with device fingerprints. About 800 events per second on the main topic, generated by my synthetic data tool.

My goal was four features for a fraud model:

1. User's transaction count in the last 30 minutes.
2. Ratio of declined to approved transactions over the last hour.
3. Number of distinct merchants the user has hit in the last 24 hours.
4. Time since the user's last chargeback.

Standard fraud feature stuff. Nothing exotic.

## Hour one: getting the source registered

I ran the Docker Compose dev setup. This was the fastest part of the whole afternoon. About four minutes from `docker compose up` to a working control plane. Defining the source was about a dozen lines of Python:

```python
from tessera import Source, Schema, Field, Timestamp

transactions = Source(
    name="transactions",
    connector="kafka",
    topic="prod.transactions",
    schema=Schema([
        Field("user_id", "string"),
        Field("amount", "float64"),
        Field("merchant_id", "string"),
        Field("status", "string"),
        Field("event_time", Timestamp()),
    ]),
    event_time_field="event_time",
    watermark_delay="30s",
)
```

`tessera apply sources/transactions.py` and it picked up the topic immediately. The CLI feedback is good. It tells you the consumer group it registered, the partitions it sees, and the current event-time watermark.

## Hour two: features one and two

The count and the decline ratio were trivial. Both compile cleanly, both materialize to Redis in real time. I had values flowing within fifteen minutes of writing the definitions.

```python
@feature(name="user_decline_ratio_1h", entity="user_id", ttl="2h")
def user_decline_ratio_1h():
    return (
        transactions
        .group_by("user_id")
        .window(window.sliding(hours=1, slide_minutes=5))
        .agg(
            declined="sum(case when status='declined' then 1 else 0 end)",
            approved="sum(case when status='approved' then 1 else 0 end)",
        )
        .select(ratio="declined / nullif(approved + declined, 0)")
    )
```

I will note that I had to look up the exact syntax for the conditional aggregate twice. The Mosaic reference docs are correct but they are organized for people who already know what they are looking for.

## Hour three: feature three (the annoying one)

Distinct merchants over 24 hours sounds easy. It is not.

The natural Mosaic spelling is `.agg(distinct_merchants="count(distinct merchant_id)")` and that works in the backfill path but does not compile to a streaming-safe operator. The error is clear enough (`UNSUPPORTED_OPERATOR_IN_STREAMING`) and the docs do call this out, but it took me about twenty minutes to find the recommended workaround, which is to use the `tessera.agg.approx_distinct` HyperLogLog primitive.

This is the kind of thing the docs honestly should hit you in the face with. "Approximate" is fine for fraud (we tolerate single-digit error on cardinality), but if I had been computing something where exact counts matter, I would have been stuck.

## Hour four: feature four and a small disaster

`Time since last chargeback` requires a temporal join. The chargebacks topic is sparse, so most users have no chargeback ever, and I want a feature that returns either a duration or null.

Mosaic has `.join_as_of`, which is exactly what I needed. The syntax is reasonable. What got me was that I forgot to set the watermark policy on the chargebacks source, so Tessera was waiting for late chargebacks indefinitely and the feature value never advanced. The diagnostic was buried in the Flink job logs, not surfaced through the CLI. I lost about thirty minutes to this.

I have filed a ticket. The response from their engineering team was, paraphrasing, "yeah we know, watermark diagnostics surfacing is on the 2.2 plan." Fine. I will live.

## What I actually liked

The lineage view is good. After four hours of work I had a graph showing my four features hanging off the three sources, with version numbers, materialization timestamps, and per-feature staleness. I could screenshot it and put it in a slide deck for my team and it would just look professional. That sounds shallow but it is genuinely a thing other feature stores get wrong.

The shadow serving path is also legitimately useful. I bumped the decline ratio window from one hour to forty-five minutes and ran the new version as a shadow for an hour. Tessera produced a comparison report showing the value distribution shift and the per-user delta. This is the kind of thing I would have had to script myself elsewhere.

## What I did not like

The Python UDF restriction. I get why. I also do not care why, because the third time you have to rewrite a function as a series of stdlib calls you stop caring about the elegance of the architecture.

The Spark adapter for backfills above 2 TB. I did not hit this in my evaluation (synthetic data, small) but I have a colleague at a larger shop who has, and his summary was "it works but you will hate it." I will trust him.

## Would I roll it out at work?

Probably. The semantic parity story is real and I have been burned enough by training-serving skew to want what they are selling. I would push hard on whatever 2.1 does for the UDF problem before signing a contract.

Total cost of the afternoon: about $11 in Confluent Cloud egress and one slightly cold coffee. Worth it.
